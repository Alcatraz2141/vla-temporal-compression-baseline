If you know LLMs

From text tokens to robot actions

If you understand GPT-style transformers, you already understand 70% of this project. Here's the mapping:

LLM: Token embeddings
VLA: Image features from ResNet18

Instead of looking up a word in a vocabulary table, each timestep's camera image is passed through a CNN (ResNet18) to produce a 512-dim feature vector, then projected to 256-dim.

LLM: Positional encoding
VLA: Learned position embedding

Same idea. Each of the 8 recent timesteps gets a learned positional embedding added so the transformer knows temporal order.

LLM: "Next token prediction"
VLA: "Next 4 actions prediction"

Instead of predicting the next word, the model predicts the next 4 action steps (each 7-dimensional: 3D position + 3D orientation + gripper open/close).

LLM: Cross-entropy loss
VLA: MSE loss on continuous actions

No vocabulary to classify over. Actions are continuous floats, so we use mean squared error. The gripper dimension gets 5x weight because it's critical.

LLM: Context window
VLA: Observation history (8 frames)

The model sees the last 8 camera frames. That's its "context window." Anything older is either discarded (sliding window) or compressed (event-gated memory).

LLM: Prompt / system message
VLA: Language task instruction

"Pick up the book and place it in the caddy" is hashed to an embedding ID and added to every token. It's the "system prompt" telling the robot what task to do.

LLM: Autoregressive inference
VLA: Closed-loop rollout

At test time, the model predicts 4 actions, executes 1, gets a new camera image, and predicts again. Like generating tokens one at a time, but each "token" moves a physical robot arm.

LLM: Hallucination / drift
VLA: Compounding error

The killer problem. After 20 steps, the robot is in a state it never saw during training. Every subsequent prediction gets worse. This is why 0% online success with low offline error is possible.

The key difference from LLMs

In an LLM, a bad token is just a bad token—the next prediction starts from the same clean context. In a robot, a bad action physically moves the arm to a new state. The next observation is now from a state the model never trained on. Errors compound geometrically. This is the fundamental challenge of behavioral cloning.

Training (Offline)

Supervised learning on recorded expert demonstrations. The model sees (images, states, actions) tuples from HDF5 files and learns to predict the expert's actions via MSE loss. The robot never actually moves during training.

Evaluation: Offline

Feed held-out episodes through the model, measure MSE and MAE between predicted and expert actions. This tells you if the model learned the mapping, but NOT if it can actually complete tasks.

Evaluation: Online (Rollout)

Load the model into the LIBERO MuJoCo simulator. The model controls the robot for up to 300 steps. Success = task completed (book in caddy, etc). This is the real test and where everything fails.

How the gate works

For each chunk of 8 older timesteps:

  1. Attention pooling: A learnable query vector attends over the chunk to produce a single summary token
  2. Delta computation: How much did vision/action/state features change within this chunk? High delta = something interesting happened
  3. Age: Linear 0→1 encoding of how old this chunk is
  4. Gate score: sigmoid(MLP([delta_visual, delta_action, delta_state, age]))
  5. Top-K selection: Keep the 8 highest-scoring memory tokens
The intuition

Imagine you're picking up a book. Most of the "reaching toward the book" frames are redundant—the arm is just moving slowly through space. But the frame where you actually grasp the book? That's a critical event. The gate should learn to score "high delta in gripper + position change" chunks higher than "arm slowly moving" chunks.

Ablation Variants

Age-Gated Memory

Same architecture but the gate ignores content deltas and just uses age:

# gate_type = "age_based" gate_score = age.clamp_min(1/chunk_count) # Recent chunks always score higher # No learning, no event awareness

Tests whether the event-detection matters or if simple recency is enough.

Concat Query (vs Cross-Attention)

Instead of the recent encoder cross-attending to memory tokens, just concatenate them:

# query_type = "concat" tokens = cat([memory_tokens, recent_tokens]) encoded = transformer(tokens) pooled = encoded[-1]

Tests whether the cross-attention mechanism adds value over simple concatenation.

Scale mismatch problem

Position deltas are ~0.01–0.05, but gripper is always -1 or +1. The MSE loss treats all dimensions equally by default, so the gripper (which has 100x larger magnitude) dominates the loss. The 5x gripper weight makes this worse, not better—the model learns to predict gripper well but position/orientation poorly. The fix is per-dimension z-score normalization before training.

Debugging journey

Six bugs that invalidated all results

These bugs were discovered during the validation process. Each one would have produced misleading results if left unfixed.

Bug #1 — Severe
Undertraining: 96.2% of data never seen

The config sampled only 1,952 windows per epoch out of ~50,000 possible. After 10 epochs, only 3.8% of available training windows were ever visited. The model had barely started learning.

Bug #2 — Severe
Temporal mismatch: never trained on timestep 0

Training sampled t from [1, N-H], but rollout always starts at t=0. The model never learned what the initial scene looks like. Fixed to sample from [0, N-H].

Bug #3 — Severe
Action history leakage: seeing the future

The "previous actions" input included the current target action. During training the model could peek at what it was supposed to predict. At rollout time this information doesn't exist, so predictions were random. Fixed: prev_actions[t] = actions[t-1], first slot = zeros.

Bug #4 — Moderate
Language mismatch: task name != filename

Training used filenames with _demo suffix as task names. Rollout used clean names from the LIBERO API. Different hash = different embedding = the model thought it was a different task entirely.

Bug #5 — Minor
Stale action chunks: executing 4 steps before replanning

The rollout executed all 4 predicted actions before getting a new observation. On a 300-step task, that's only 75 replan points. Fixed to replan every step (execute_horizon=1).

Bug #6 — Design flaw
Gripper as continuous regression instead of binary classification

The gripper is always -1 (close) or +1 (open) in demos. Using MSE regression means the model can predict 0.3, which is neither open nor closed. Should be a separate binary cross-entropy head with threshold at 0.

All six bugs were fixed

The corrected pipeline was validated: expert demo replay now succeeds in simulation, temporal alignment is correct, action history is properly shifted, language names match. But the policy still gets 0% task success because the bugs were necessary but not sufficient fixes.

Root Cause Analysis

Why 0% online success?

Multiple factors contribute. Here they are ranked by likely impact:

Critical Compounding error in behavioral cloning

This is the fundamental limitation. BC learns P(action | observation) from expert data. But at test time, the robot's small errors push it into states never seen during training. Each subsequent prediction gets worse.

Scale of the problem: LIBERO-Long tasks require ~300 steps. Even with 99% per-step accuracy, the probability of a perfect trajectory is 0.99300 = 5%. With realistic 95% accuracy: 0.95300 ≈ 0%.

What helps: DAgger (online correction data), diffusion policy, or action chunking with receding horizon (ACT).

Critical No action normalization

Position deltas (~0.01–0.05) and gripper (-1/+1) have wildly different scales. Without per-dimension z-score normalization, the loss and optimizer treat them identically.

# What should happen (doesn't exist yet): mean, std = compute_dataset_stats() # per-dimension normalized_action = (action - mean) / std # for training raw_action = prediction * std + mean # at rollout

This is table-stakes for any manipulation paper. Its absence is a silent performance killer.

High Gripper treated as continuous regression

The gripper is always exactly -1 (close) or +1 (open). Using MSE lets the model predict 0.3 or -0.5—values that map to neither a firm grasp nor a clean release. The discrete_gripper flag in rollout thresholds at 0, but by then the model has already been trained wrong.

Fix: Separate BCE head for gripper dimension. Predict a logit, threshold at 0 for hard -1/+1 at inference.

High ResNet18 from scratch = poor visual features

Although pretrained_vision: true is set, the ResNet18 is fine-tuned during training with the same learning rate as everything else. With only 50K windows, the visual encoder overfits to the training demos without learning generalizable visual features.

Papers like R3M and MVP show that frozen pretrained encoders (even simple ImageNet ViT-B) dramatically improve sample efficiency on LIBERO.

Medium Task horizon is extremely long

LIBERO-Long tasks require 200–300 steps. This is one of the hardest benchmarks for BC. ACT and Diffusion Policy papers typically evaluate on tasks with 50–100 step horizons. Going straight to LIBERO-Long without validating on shorter tasks (LIBERO-Spatial, LIBERO-Object) means you can't tell if the model fundamentally can't do BC or just can't handle long horizons.

Medium State representation mismatch (potential)

Training reads states from HDF5: ee_pos + ee_ori + gripper_states. Rollout constructs states from sim: robot0_eef_pos + quat_to_axis_angle(robot0_eef_quat) + gripper_qpos. If the orientation representations differ (Euler vs axis-angle vs quaternion), the state distribution shifts enough to degrade closed-loop performance even if magnitudes look similar.

Low Model capacity

A 4-layer, 4-head, 256-dim transformer on top of ResNet18 is small. ACT uses a CVAE + larger transformer. Diffusion Policy uses a UNet denoiser. But model capacity shouldn't be the first thing to fix—the data/normalization issues are more impactful.

Path Forward

What to fix, in priority order

Step 0 — In progress
Train event-gated to 50 epochs (fair comparison)

Step 1 — Quick validation
Test on shorter tasks (LIBERO-Spatial/Object)

30-minute diagnostic. If BC works on 50-step tasks, the problem is horizon length specifically. If it fails there too, something more fundamental is wrong.

Step 2 — Essential fix
Add per-dimension action normalization

Compute mean/std over training set. Z-score normalize inputs, denormalize predictions. 30 lines of code, potentially large impact.

Step 3 — Essential fix
Separate BCE head for gripper

Replace MSE on dim 6 with binary cross-entropy. Predict logit, threshold at inference. Standard in ACT and most recent papers.

Step 4 — High impact
Freeze the visual encoder

Use ImageNet-pretrained ResNet18 (or better, R3M/ViT) with weights frozen. Only train the transformer + action head. Faster training, better generalization.

Step 5 — The real paper
Run sliding-window vs event-gated with all fixes

Same encoder, same normalization, same gripper head, same epochs. The only variable is the memory architecture. This is the paper's core comparison.

Step 6 — Ablations
Gate ablations + visualization

Age-gated (no event detection), concat query (no cross-attention). Visualize which timesteps get high gate scores. This is what makes reviewers believe the method.

Complete model forward pass

EventGatedMemoryVLA: step by step

def forward(self, recent_obs, recent_actions, recent_states, older_obs, older_actions, older_states, ...): # STEP 1: Encode recent context (same as sliding window) recent_tokens = _encode_steps(recent_obs, recent_actions, recent_states) # recent_obs: [B, 8, 3, 128, 128] → ResNet18 → proj → [B, 8, 256] # recent_actions: [B, 8, 7] → MLP → [B, 8, 256] # recent_states: [B, 8, 8] → MLP → [B, 8, 256] # cat → [B, 8, 768] → fusion MLP → [B, 8, 256] # STEP 2: Encode older context older_tokens, older_vis, older_act, older_st = _encode_steps( older_obs, older_actions, older_states) # older_obs: [B, 64, 3, 128, 128] → same pipeline → [B, 64, 256] # STEP 3: Chunk older context into groups of 8 # [B, 64, 256] → [B, 8_chunks, 8_per_chunk, 256] # STEP 4: Attention-pool each chunk to a single summary # learnable query dot-product with chunk tokens # softmax → weighted sum → [B, 8_chunks, 256] # STEP 5: Compute event signals per chunk delta_visual = mean_abs_diff(visual_features_in_chunk) # [B, 8, 256] delta_action = mean_abs_diff(action_features_in_chunk) # [B, 8, 256] delta_state = mean_abs_diff(state_features_in_chunk) # [B, 8, 256] age = linspace(0, 1, 8_chunks) # [B, 8, 1] # STEP 6: Gate scoring gate_input = cat([delta_visual, delta_action, delta_state, age]) # [B, 8, 769] gate_score = sigmoid(MLP(gate_input)) # [B, 8, 1] memory_tokens = summaries * gate_score # [B, 8, 256] # STEP 7: Cross-attention (query_type="cross_attention") recent_encoded = transformer(recent_tokens) # [B, 8, 256] query = recent_encoded[:, -1:] # last timestep: [B, 1, 256] attended = cross_attn(query, memory_tokens, memory_tokens) # [B, 1, 256] pooled = query + attended # residual connection # STEP 8: Predict actions output = action_head(pooled) # [B, 256] → [B, 28] → [B, 4, 7]

Reference

Glossary for LLM people

TermWhat it means
VLAVision-Language-Action model. Takes images + language instruction, outputs robot actions. Like a multimodal LLM that outputs motor commands instead of text.
BCBehavioral Cloning. Supervised learning on expert demonstrations. The simplest imitation learning approach. Analogous to training an LLM on a fixed corpus with no RLHF.
LIBEROA benchmark suite of robot manipulation tasks in MuJoCo simulation. LIBERO-Long has 10 long-horizon tasks (~300 steps each).
End-effector (EEF)The robot's "hand" — the gripper at the end of the arm. Actions control its position and orientation.
Delta actionsInstead of absolute positions, the model predicts changes: "move 0.02m right, 0.01m forward." Like velocity rather than position.
Action chunkingPredicting multiple future actions at once (we predict 4). Like generating 4 tokens in one forward pass.
RolloutRunning the trained model in the simulator in a closed loop. The model's outputs actually control the robot. This is the real evaluation.
Compounding errorSmall mistakes accumulate because each bad action changes the robot's state. After 20 bad actions, the robot is somewhere it's never seen before.
DAggerDataset Aggregation. Run the model, when it makes mistakes, have the expert correct it, add corrections to training data. Fixes compounding error but requires online expert access.
ACTAction Chunking with Transformers. A strong BC baseline that uses a CVAE + transformer for action chunking. Often gets 30-50% success on similar tasks.
Diffusion PolicyGenerates actions by iterative denoising (like image diffusion models). Handles multimodal action distributions well. Current state-of-the-art for BC.
R3M / MVPPretrained visual representations for robotics. Like using a pretrained language model instead of training word embeddings from scratch.
